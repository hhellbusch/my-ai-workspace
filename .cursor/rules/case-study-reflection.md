# Case Study Reflection

When completing a non-trivial piece of work — building a tool, solving a multi-step problem, creating a new workflow pattern, integrating a new concept — briefly consider:

1. **Does this work demonstrate a pattern that connects to an existing essay in `docs/`?** (e.g., the sparring integration extends the meta-development pattern from `building-a-research-skill.md`)
2. **Could this work become its own case study?** (e.g., a session that produced a novel workflow, solved an interesting problem, or revealed a non-obvious insight)
3. **Does this work validate or challenge a claim made in an existing doc?** (e.g., work that proves or disproves an assertion in `the-shift.md` or `ego-ai-and-the-zen-antidote.md`)

## When to Capture

- After completing a multi-step task that produced reusable artifacts
- After a session where the *process* was as interesting as the *output*
- When you notice the current work mirrors a pattern already documented in the essay library
- When the work reveals a gap or limitation in an existing essay's claims

## How to Capture

Add a seed to `BACKLOG.md` under **Ideas** with the `Case study:` title prefix:

```markdown
### Case study: [descriptive title]
- **Product:** docs
- **Context:** [1-2 sentences: what happened, what pattern it demonstrates, which existing essays it connects to]
- **Links:** [paths to artifacts created, existing essays it relates to]
- **Added:** [date]
```

Follow the existing `backlog-capture.md` convention: commit the backlog change immediately with a `backlog:` prefix.

Mention it briefly to the user: "Added a case study idea to the backlog: [title]." The user decides later whether to pursue it.

## What This Is Not

- Not a gate — don't block work to write case study notes
- Not a prompt to the user — don't ask "should this be a case study?" during work; just capture the seed
- Not a planning document — case study seeds are backlog items, not briefs or roadmaps
- Not for trivial work — don't capture every config change or file rename
