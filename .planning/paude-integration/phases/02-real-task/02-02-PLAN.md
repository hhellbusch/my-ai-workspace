<task type="auto">
  <name>Task 1: Write the Paude task spec</name>
  <files>.planning/paude-integration/task-specs/02-02-spec.txt</files>
  <action>
    Write the task spec to `.planning/paude-integration/task-specs/02-02-spec.txt` based on the
    chosen topic. The spec must pass all four checklist gates:

    1. Does NOT start with a word matching a workspace slash command (e.g., Review, Edit, Search,
       Grill, Run). Good starting verbs: Analyze, Examine, Survey, Audit, Investigate, Compare.
    2. All paths, names, and quantities are explicit — no "the notes folder", use full paths.
    3. Ends with an explicit commit instruction:
       `After completing all work, run: git add -A && git commit -m "survey: describe what changed"`
    4. Includes the AGENT-NOTES request:
       `After completing the task, write AGENT-NOTES.md in the workspace root documenting:
       what decisions did you make that the spec did not cover? What was ambiguous?
       What assumptions did you make?`

    The spec should specify:
    - Exactly which files to read as context
    - What the agent should produce (proposed edits, new sections, or a separate proposal file)
    - The output format (direct edits to existing files, or a new file with proposals)
    - The scope boundary (what NOT to change — e.g., do not modify examples/ directories)

    Keep the spec under ~300 words. Headless agents do better with tight, explicit briefs than
    long open-ended ones. Ambiguity causes stalling.

    Preferred output format: the agent edits existing files directly and commits the changes.
    This produces a diff that can be reviewed and merged or discarded.
  </action>
  <verify>
    cat .planning/paude-integration/task-specs/02-02-spec.txt
    Confirm: file exists, starts with an action verb (not a slash command keyword), ends with
    explicit git commit instruction and AGENT-NOTES request.
  </verify>
  <done>

  <action>
    Step 1 — Commit the spec file so it is available inside the container:
    ```bash
    cd /home/hhellbusch/gemini-workspace
    git add .planning/paude-integration/task-specs/02-02-spec.txt
    git commit -m "chore: add paude task spec for 02-02 ecosystem survey"
    ```

    Step 2 — Create the Paude session with a short bootstrap prompt pointing to the file.
    The bootstrap prompt contains no embedded quotes, backticks, or newlines — it just tells the
    agent where to find the real instructions:
    ```bash
    paude create paude-survey-02 --git --yolo \
      -a '-p "Read the file .planning/paude-integration/task-specs/02-02-spec.txt and execute the task described in it. Do not use any workspace slash commands. When all work is complete, commit everything."'
    ```

    Step 2 — Monitor until Idle:
    ```bash
    paude status paude-survey-02
    # Repeat every 30-60 seconds. Active = working, Idle = done or stalled.
    # If Active for >15 minutes, connect and inspect: paude connect paude-survey-02
    ```

    Step 3 — Verify a commit exists before harvesting (do NOT skip this):
    ```bash
    paude connect paude-survey-02
    # Inside the container:
    git log --oneline -3
    # If the expected commit is present, detach with Ctrl+b d and proceed to harvest.
    # If git log shows nothing new, check git status — commit manually if files are there.
    ```

    Step 4 — Harvest (do NOT reset before harvesting):
    ```bash
    paude harvest paude-survey-02 -b review/survey-02
    git diff main..review/survey-02 --stat
    ```

    Step 4 — Analyze in order:
    1. Read AGENT-NOTES.md from the harvested branch first
    2. Read the full diff: `git diff main..review/survey-02`
    3. Assess output quality against the spec intent

    Step 5 — Write findings to `.planning/paude-integration/findings/phase-2-02-findings.md`.
    The findings file must answer all Phase 2 key questions from the ROADMAP:
    - Did `.cursorrules` provide enough context for a broad-scope task without steering?
    - How does output quality compare to an interactive session?
    - What overhead did the harvest-review cycle add?
    - Did the spec quality (not the tool) determine output quality?
    - What would need to change in the spec pattern for this task type to be reliable?

    Include verbatim excerpts from AGENT-NOTES.md to support the answers.
  </action>
