<objective>
Test the complete fire-and-forget orchestration cycle: assign a task, disconnect immediately,
return when done, harvest to a PR, then reset the session and assign a second task. The goal is
to assess whether the create → assign → disconnect → status → harvest → PR → reset → reassign
cycle is ergonomic enough to justify the container overhead for solo dev work.

Purpose: Answer Phase 3 key questions — is the orchestration overhead worth it, how smooth is
the reset/reassign cycle, does `paude status` provide adequate visibility, and does a task brief
alone (without interactive steering) match interactive quality when you can't course-correct?

Output:
- Two harvested branches (one per task) and one GitHub PR
- `.planning/paude-integration/findings/phase-3-01-findings.md` — analysis of the full cycle
</objective>

    ```bash

    ```
    Note: PR quality, diff size, any obvious issues.

    Step 3 — Reset the session for Task 2:
    ```bash
    paude reset paude-orch-03
    # Wait for confirmation that reset completed.
    paude status paude-orch-03
    ```

    IMPORTANT: Reset wipes conversation history and resets the workspace. Only reset after
    harvesting — never before.

    Step 5 — Assign Task 2 (Task 2 spec was already committed in the setup task):
