  <files>.planning/paude-integration/ASSESSMENT.md</files>
  <action>
    Read all four findings files plus the BRIEF and the Yegge Gas Town reference before writing.
    The assessment must be evidence-based — every claim should trace to a specific finding,
    a specific AGENT-NOTES excerpt, or a specific observed behavior. No speculation.

    Write `.planning/paude-integration/ASSESSMENT.md` with this structure:

    ---
    # Paude Integration Evaluation — Assessment

    ## Executive Summary
    One paragraph. Keep/skip/integrate decision and the primary reason. No hedging.

    ## Evaluation Scope
    What was tested (phases 1-4), what was explicitly NOT tested (OpenShift backend, production
    deployment, team workflows), and how that scopes the conclusions.

    ## Findings by Phase
    One section per phase (2-4 — Phase 1 was mechanics, just note it completed).
    For each phase: what was tested, what was found, what it means.

    ## PAI/Kai Lens: Single Review Point or Army Overhead?
    Address directly: does the harvest workflow consolidate review behind a single point
    (analogous to asking Kai what happened), or does it recreate coordination overhead?
    Use specific evidence from the findings — not a theoretical answer.
    Reference the Yegge Gas Town architecture for comparison: how does Paude's create →
    harvest cycle compare to Gas Town's Polecat → Refinery → MR workflow?

    ## Keep / Skip / Integrate

    ### Keep: Use Cases Where Paude Adds Value
    Be specific. "Fire-and-forget documentation tasks" is better than "autonomous work."
    Each use case should reference which phase surfaced it.

    ### Skip: Use Cases Better Served by In-Session Approach
    What should stay as in-session Task subagents? Why? Evidence-based.

    ### Integrate: If Keeping, How?
    Specific integration points from the BRIEF success criteria:
    - Add `paude.json` to workspace? (yes/no, why)
    - `~/.config/paude/defaults.json` setup? (recommended settings)
    - `/paude` slash command? (yes/no, what it would do)
    - External executor pattern in meta-prompts? (yes/no, how)
    - Any workspace convention changes?

    ## Blockers and Open Questions
    What would need to be true for Paude to be more useful? What was unknown at the start
    and is still unknown after the evaluation?

    ## Decision
    One sentence. Keep and integrate, keep without integration, or skip.
    ---

    Tone: practitioner, direct, no hedging. Same voice as the RHACM troubleshooting guides.
    Length: aim for 600-1000 words — long enough to be substantive, short enough to be read.
