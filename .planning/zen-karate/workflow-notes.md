# Workflow Notes — What Worked and Why

Notes on the development workflow that emerged during this project. Captured for reuse and reflection.

---

## The Thread Development Pass

**What it is:** A single-pass assessment of every idea thread in a creative project — producing a landscape document rather than diving into any single thread.

**What it produces (per thread):**
- Thesis statement (one sentence)
- Key arguments with source citations
- Evidence inventory (what we have, categorized by source)
- Gap analysis (research gaps, personal experience gaps, logical gaps)
- Integrated spar notes (adversarial counterarguments woven in, not separate)
- Verdict: Ready to draft / Needs your voice / Needs research / Merge candidate / Kill candidate

**What it produces (cross-thread):**
- Merge candidates (threads saying the same thing)
- Honest assessment of AI-synthesized connections vs. practitioner insight
- Project-level adversarial review (does the thesis survive?)
- Open questions the pass raises

### Why It Worked

1. **Landscape before depth.** Instead of picking a thread and going deep immediately, the full pass revealed relationships and contradictions between threads that would have been invisible from inside any single thread. The thread 9 vs. 13 contradiction (no style matters / scaled karate lost its essence) was only visible by developing both.

2. **Sparring during development, not after.** The adversarial review was integrated into each thread's section rather than being a separate phase. This meant the spar challenged the argument while it was being built — like a training partner who challenges you during practice, not a critic who reviews the performance afterward. Each thread's verdict already incorporates the spar's findings.

3. **Verdicts create triage.** The five-option verdict system (ready / needs voice / needs research / merge candidate / kill candidate) turned a sprawling ideation document into a prioritized map. Instead of "16 interesting ideas," the output is "3 that survive scrutiny, 7 that need personal experience, 3 that should merge, 2 that need research, 1 that's already published."

4. **Honest ratio assessment.** The cross-thread spar forced a question most creative projects avoid: "what percentage of this is genuinely my insight vs. research compilation?" The ~20/80 finding was uncomfortable but useful — it makes clear where the work actually is (personal experience capture), rather than where it feels productive (more research, more infrastructure).

5. **Gap categorization matters.** Distinguishing between research gaps (solvable by more reading), personal experience gaps (only the author can fill), and logical gaps (the argument doesn't connect) prevents the mistake of researching your way around a gap that only personal testimony can fill.

### The Pattern

```
Research accumulation (sessions)
    → Planning evaluation (honest state assessment)
    → Thread development pass (all threads, outline depth, with sparring)
    → Landscape document (verdicts, merges, contradictions, honest ratios)
    → Prioritized next steps (fill the gaps that matter most)
```

This is distinct from the standard essay pipeline (`research → plan → draft → review → publish`) — it's a pre-pipeline assessment that determines which threads are ready for the pipeline and which need more foundation work first.

### What It Doesn't Do

- Doesn't produce finished prose — scaffolding, not essays
- Doesn't replace personal input — makes the gaps impossible to ignore
- Doesn't create new research — flags what's needed
- Doesn't guarantee quality — a well-scaffolded bad argument is still a bad argument

## The Research → Evaluate → Develop Cycle

The broader workflow pattern across this session:

1. **Deep research on specific figures** (Rika Usami, expanded Inoue) — not broad research, but targeted depth into lineage and teaching philosophy
2. **Honest project evaluation** — not "update the roadmap" but genuinely reassessing sequencing, progress, and what the research changed. This included acknowledging that Phase 1 was ~40% not ~70% because structural scaffolding ≠ experiential depth.
3. **Full thread development with sparring** — the landscape pass above
4. **Workflow reflection** (this document) — capturing what worked while it's fresh

The key discipline: **evaluating before developing.** The temptation after good research is to immediately start writing. The evaluation step caught the essay sequencing issue (Way Is in Training should lead, not the applied essay) and the Phase 1 overestimate. Without that pause, the development pass would have been building on wrong assumptions.

## Connection to Existing Meta-System

This workflow engages several existing tools:
- **`/spar` patterns** — the adversarial framework was used throughout the development pass
- **Cross-linking conventions** — source citations tie each argument to cached research
- **Backlog discipline** — verdicts feed back into backlog prioritization
- **Shoshin principle** — the honest ratio assessment and the phase progress correction both reflect "check what you think you know"

What it adds that the meta-system didn't have:
- A **pre-pipeline landscape assessment** format — the thread-development document as a reusable artifact type
- **Verdict-based triage** for creative/essay work — a structured way to sort ideas before committing to drafting
- **Integrated sparring** as a development mode, not just a review mode

---

## When to Use This Workflow

**Good fit:**
- Creative projects with many possible threads/essays/features
- Projects where research has accumulated but writing hasn't started
- Projects where personal vs. AI contribution ratio matters
- Before committing to a drafting order or essay sequence

**Not needed for:**
- Single-thread work with a clear thesis
- Technical documentation where the facts determine the structure
- Work where the research and writing happen in the same pass
