# Git-Driven Cluster Labels

Cluster group memberships and app opt-in/out labels are managed **in Git** and
enforced on the hub cluster via RHACM Policies. This eliminates manual
`oc label` commands and ensures all cluster configuration is version-controlled.

## How It Works

1. Each cluster's `cluster.yaml` defines the intended labels
2. An RHACM Policy (generated by Helm) reads `cluster.yaml` and enforces those
   labels on the `ManagedCluster` resource
3. RHACM propagates `ManagedCluster` labels to the ArgoCD cluster secret
4. ArgoCD ApplicationSet generators read labels from the cluster secret

```
Git (cluster.yaml) ──▶ RHACM Policy ──▶ ManagedCluster labels ──▶ ArgoCD cluster secret
```

All label changes go through the same promotion pipeline:
`main → release/dev → release/staging → release/production`

## The Cluster Label Sync Chart

`hub/rhacm/cluster-labels/` is a Helm chart deployed by ArgoCD on the hub.
It reads every cluster directory under `clusters/` and generates one
RHACM Policy per cluster that enforces the labels defined in `cluster.yaml`.

This chart is deployed automatically as part of the hub bootstrap — no
manual intervention needed. When you add or change labels in `cluster.yaml`
and merge the PR, RHACM picks up the policy change and updates the labels.

## Adding or Changing Labels

1. Edit `clusters/<name>/cluster.yaml` — update the `managedClusterLabels` section
2. Commit and push (promotion pipeline handles the rest)
3. After merge, RHACM Policy enforces the new labels on the ManagedCluster
4. ArgoCD ApplicationSets react to the label change automatically
