# Convergence with gitops-standards-repo-template

> **Status: Aspirational / working document**
>
> This document sketches an integration path between this Helm-centric pattern and the Kustomize-centric [redhat-cop/gitops-standards-repo-template](https://github.com/redhat-cop/gitops-standards-repo-template). The approach described here requires validation with the CoP team before it can be considered production guidance. It is recorded here to capture the direction of thinking and provide a starting point for that conversation.

---

## The integration model

The most pragmatic convergence path is: **choose engine per component, share the inventory**.

Keep `clusters.yaml` as the unified fleet inventory. Let each component declare its own `sourceType` — Helm (default) or Kustomize. The `hub-clusters` chart emits the correct ArgoCD Application source block based on that flag.

```
clusters.yaml               ← unified fleet inventory (unchanged)
components/
  cert-manager/             ← Helm chart  (sourceType: helm — default)
  nmstate/                  ← Kustomize component  (sourceType: kustomize)
    base/
      kustomization.yaml
    overlays/               ← optional group-specific patches
      virt-enabled/
        kustomization.yaml
groups/
  all/values.yaml
    # component-all:
    #   apps:
    #     cert-manager: {enabled: true}                    # sourceType: helm (default)
    #     nmstate:       {enabled: true, sourceType: kustomize}
clusters/
  site-dc1/values.yaml      ← Helm overrides only; Kustomize patches live in components/
```

## What is implemented today

`hub-clusters` supports a `sourceType` field on each app entry:

```yaml
# groups/all/values.yaml
component-all:
  apps:
    cert-manager:
      enabled: true
      # sourceType: helm   ← default, no field needed
    nmstate:
      enabled: true
      sourceType: kustomize
```

For `sourceType: helm` (default): current behaviour — full `mustMergeOverwrite` chain injected into `spec.source.helm.values`.

For `sourceType: kustomize`: ArgoCD uses `kustomize build` on the component path. Cluster identity from `clusters.yaml` is injected as `commonLabels` and `commonAnnotations`:

```yaml
kustomize:
  commonLabels:
    cluster-name: "site-dc1"
    cluster-environment: "production"
    cluster-region: "us-east"
    cluster-hub: "prod-a"
  commonAnnotations:
    cluster-vault: '{"server":"https://vault.prod-a.example.com","clusterSecretStoreName":"vault-backend"}'
    cluster-monitoring: '{"remoteWriteEndpoint":"https://mimir.prod-a.example.com/api/v1/push"}'
```

---

## The `commonAnnotations` limitation

Kustomize `commonLabels` are flat string key-value pairs — suitable for cluster identity metadata but not for complex nested configuration that operators or controllers consume. `commonAnnotations` carries JSON-encoded strings for nested attributes, which is awkward to consume in Kustomize patches.

This is a fundamental constraint of single-source Kustomize: the component path receives only what ArgoCD can inject through standard Kustomize mechanisms.

### What this means in practice

A Kustomize component that needs `.Values.cluster.vault.server` in a Helm chart cannot access `cluster-vault: '{"server":"..."}'` from a commonAnnotation without custom tooling. If the component needs rich per-cluster configuration, either:

1. Keep it as a Helm component (`sourceType: helm`) and use the full values chain, or
2. Use **ArgoCD multi-source** (ArgoCD 2.6+) to supply a second source with cluster-specific patches alongside the Kustomize base

### ArgoCD multi-source (future extension — not yet implemented)

```yaml
# In hub-clusters template — sketch of multi-source support
spec:
  sources:
    - repoURL: {{ $.Values.source.repoURL }}
      targetRevision: {{ $targetRevision }}
      path: components/nmstate          # Kustomize base
    - repoURL: {{ $.Values.source.repoURL }}
      targetRevision: {{ $targetRevision }}
      path: clusters/{{ $clusterName }}/patches/nmstate   # cluster-specific patches
```

The cluster-specific patch directory would be generated by CI from `clusters.yaml` — a natural extension of the existing `hub-bootstrap` rendering step. This is not yet implemented.

## Migration path from gitops-standards-repo-template

> **Caution:** Step 3 below (adding `clusters.yaml`) is a breaking change to existing Application generators if your current setup uses cluster directories as the generator source. Plan the generator transition explicitly — running both the old generator and `hub-clusters` simultaneously during migration is possible but requires care to avoid duplicate Application objects.

1. **Keep your existing Kustomize components unchanged.** Add `sourceType: kustomize` to their entries in `groups/all/values.yaml`.
2. **Migrate one component at a time.** Convert a Kustomize component to a Helm chart by adding a `Chart.yaml` and `templates/` folder and removing `sourceType: kustomize`.
3. **Add `clusters.yaml`.** Centralise the cluster inventory that was previously implicit in cluster directories or spread across individual kustomization overlays. This is the highest-value change — it makes the fleet topology inspectable and auditable in one file. **This requires retiring the existing ApplicationSet or Application generator and replacing it with `hub-clusters`.**
4. **Adopt `hub-clusters` as the Application generator.** Replace the existing generator with `hub-clusters`. Kustomize components continue to work with `sourceType: kustomize`; Helm components use the default path.

## Open questions for the team

These need resolution before this guidance can be called production-ready:

- [ ] How do group-level Kustomize overlays interact with the `hub-clusters` generator? Currently a Kustomize component's overlays live inside `components/<name>/overlays/` — how does a group signal "use this overlay for my clusters" without per-cluster patch files?
- [ ] What is the transition strategy for an existing fleet using gitops-standards? How do you run `hub-clusters` alongside an existing ApplicationSet without duplicate Applications?
- [ ] Is `commonAnnotations` with JSON strings acceptable for the use cases we care about, or do we need multi-source from the start?
- [ ] Should `clusters.yaml` addition be automated (e.g. a `make add-cluster` target) to reduce the merge conflict risk on large teams?

## What stays the same across both patterns

| Aspect | gitops-standards-repo-template | This pattern |
|---|---|---|
| Folder layout | `components/`, `groups/`, `clusters/` | Same |
| Composable groups | Cluster belongs to multiple groups | Same |
| Component isolation | Each component is a separate Application | Same |
| Target platform | OpenShift / Kubernetes | Same |
